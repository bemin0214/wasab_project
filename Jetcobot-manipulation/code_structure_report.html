Jetcobot Manipulation 코드 구조 분석

노트북 서버와 Raspberry Pi/Jetcobot arm 클라이언트가 분리된 구조로, 서버는 YOLO 검출과 2D→3D 파지 계획을 담당하고 arm 쪽은 카메라 캡처, 안전 검증, MyCobot 제어를 담당한다.

개요 디렉터리 동작 흐름 서버 Arm 클라이언트 API 설정 주의점

1. 전체 개요

Laptop Server Jetcobot-manipulation_server

FastAPI 기반 서버다. 시작 시 YOLO 모델과 카메라 intrinsic, hand-eye 캘리브레이션을 로드한다. arm 클라이언트가 올린 이미지와 현재 Flange pose를 받아 검출과 좌표 변환을 수행하고 최종 flange_command를 반환한다.

Raspberry Pi / Jetcobot Jetcobot-manipulation_arm

카메라를 열고 MyCobot280을 제어한다. 서버 연결 확인, 카메라 미리보기 업로드, 원격 버튼 폴링, 파지 계획 요청, Pi-side 안전 검증, 로봇팔/그리퍼 동작을 수행한다.

중요한 설계 원칙은 무거운 인식/기하 계산은 노트북, 실제 로봇 명령과 최종 안전 게이트는 Pi에 둔다는 점이다.

3. 실행 및 데이터 흐름

기본 실행 순서

  1. 노트북에서 Jetcobot-manipulation_server/run_server.py 실행.
  2. 서버가 0.0.0.0:8000에서 FastAPI 앱을 띄우고 YOLO/캘리브레이션을 로드.
  3. Pi/Jetcobot에서 Jetcobot-manipulation_arm/run_client.py 실행.
  4. arm 클라이언트가 /health로 서버를 확인하고 카메라, MyCobot을 초기화.
  5. 카메라 producer thread가 최신 프레임 buffer를 갱신하고, 스트림 스레드가 상태 오버레이를 그려 POST /camera-frame으로 업로드.
  6. 노트북 브라우저는 /camera-view에서 MJPEG 스트림 /camera-frame/stream.mjpg를 표시.

G 버튼 또는 g 키 파지 흐름

Pi 카메라/로봇 상태
fresh frame + current flange coords
→
서버 YOLO + Geometry
bbox center → ray → Base plane → flange_command
→
Pi 안전 검증 + 실행
workspace gate → send_coords → gripper close
POST /v1/grasp-plan
  form image: JPEG frame
  form robot_state: { request_id, flange_coords: [x,y,z,rx,ry,rz] }

response
  status: ok | not_found
  detection: selected object
  plan.flange_command: [x,y,z,rx,ry,rz]

YOLO 검출 후 좌표 계산 그림

로봇은 단순히 정해진 Z만큼 내려가는 것이 아니다. YOLO bbox 중심에서 나온 camera ray를 현재 Flange pose, camera intrinsic, hand-eye calibration으로 Base 좌표계 ray로 바꾼 뒤, object_plane_z_base_mm + toothbrush_target_z_offset_mm 평면과 만나는 TCP 목표를 계산한다. 이후 tcp_offset_flange_to_tcp_mm를 보정해 최종 flange_command 전체 좌표 [x,y,z,rx,ry,rz]를 만든다.

4. 노트북 서버 코드 설명

파일역할핵심 포인트
run_server.py FastAPI 서버 실행 진입점 server_config.ini의 host/port를 사용해 uvicorn 실행.
app/settings.py 서버 설정 로드 YOLO 모델 경로, device, confidence, 캘리브레이션 파일, 이미지 크기 제한을 dataclass로 제공.
app/main.py API, YOLO inference, camera-view, 원격 명령 큐 시작 시 모델/캘리브레이션을 한 번 로드. state.inference_lock으로 YOLO 동시 접근 보호.
app/geometry.py 2D 검출 결과를 3D 파지 명령으로 변환 pixel center를 undistort한 ray로 만들고, hand-eye ^fT_c와 현재 flange pose를 이용해 Base 좌표계의 TCP 목표를 계산.

서버 상태 객체

ServerState는 YOLO 모델, 캘리브레이션, inference lock, 최신 카메라 JPEG, MJPEG 표시용 metadata, 원격 명령 queue를 들고 있다. 이 구조 덕분에 API 함수들이 같은 런타임 상태를 공유한다.

YOLO 검출과 파지 계획

_run_inference()는 Ultralytics YOLO를 실행하고, confidence 기준으로 detection 목록을 만든다. _create_grasp_plan()은 가장 높은 confidence detection 하나를 선택해 compute_grasp_plan()에 넘긴다.

YOLO bbox [x1,y1,x2,y2]
→ bbox midpoint (u,v)
→ cv2.undistortPoints()로 camera ray
→ ^bT_c = ^bT_f @ ^fT_c
→ object_plane_z_base_mm 평면과 ray 교차
→ tcp_offset_flange_to_tcp_mm 보정
→ flange_command 반환

카메라 화면

arm 클라이언트가 POST /camera-frame으로 올린 JPEG bytes를 서버가 재인코딩 없이 저장하고, 브라우저는 /camera-frame/stream.mjpg로 MJPEG 스트림을 본다. 상태 텍스트는 /camera-frame/status를 1초마다 조회한다.

원격 버튼

/camera-view에는 G/P/Q/R/T/W/X 버튼이 있고, 버튼 클릭은 POST /robot-command/{command}로 서버 queue에 명령을 넣는다. arm 클라이언트는 GET /robot-command/next를 폴링해 키 입력과 같은 로직으로 처리한다.

5. Arm 클라이언트 코드 설명

파일역할핵심 포인트
run_client.py Pi 실행 진입점 robot_client.main.main() 호출.
robot_client/config.py 클라이언트 설정 로드 서버 URL, 카메라, 스트리밍, 원격 제어, 로봇 속도, 안전 범위 등을 읽는다. camera_id는 숫자 또는 /dev/videoX 경로 모두 가능.
robot_client/api_client.py 서버 HTTP 통신 /health, /camera-frame, /robot-command/next, /v1/grasp-plan 호출 담당. 네트워크 오류 시 배포 상황에 맞는 힌트를 제공.
robot_client/main.py 메인 제어 루프 카메라 open, 서버 preflight, home 이동, 원격 명령 처리, G/P/Q/R/T/W/X 동작 실행. 카메라 캡처 producer와 스트리밍은 별도 스레드로 계속 동작.
robot_client/robot_controller.py MyCobot280 제어 Flange mode 설정, 좌표 읽기, send_coords, target 도달 대기, 그리퍼 open/close, throw 모드 실행.

클라이언트 메인 루프

main.py는 시작 시 서버 health check, 카메라 해상도 검증, MyCobot 초기화, home/open 동작을 수행한다. 이후 루프에서는 로컬 키 또는 서버 원격 명령을 같은 key 값으로 정규화해 처리한다.

  • g: 현재 flange pose와 fresh frame을 서버로 보내 grasp plan 요청.
  • p: 현재 Flange pose를 읽어 콘솔과 화면 overlay 상태에 표시.
  • q: 그리퍼 close/open 토글. 물체를 수동으로 잡거나 놓을 때 사용.
  • r: home 주변의 제한된 범위 안에서 안전 랜덤 Flange pose로 이동.
  • t: grasp 이후 throw 동작 시작.
  • w: 현재 동작을 중단하고 gripper 상태와 관계없이 home 위치로 복귀.
  • x: throw 중이면 abort/stop 후 클라이언트 종료.

카메라 스트리밍 스레드

카메라 화면이 G 동작 중 멈추지 않도록 camera_capture_worker()가 카메라를 단독으로 읽고 최신 frame buffer를 갱신한다. camera_stream_worker()는 이 buffer의 복사본을 서버로 업로드한다. 메인 루프가 YOLO 응답이나 로봇 이동 완료를 기다려도 캡처와 스트리밍은 계속 진행된다.

안전 검증

서버가 보낸 flange_command는 Pi에서 다시 검증된다. X/Y/Z 범위, Euler 각도 절대값, 숫자 유효성, 리스트 길이를 확인한 뒤에만 실제 로봇 명령을 보낸다. 이 final safety gate는 서버 계산 오류나 네트워크 데이터 오류가 실제 움직임으로 이어지는 것을 막는 핵심 방어선이다.

7. 설정 파일 핵심 항목

server_config.ini

  • [server] host = 0.0.0.0: Pi에서 노트북 LAN IP로 접속 가능하게 유지.
  • port = 8000: FastAPI 서버 포트.
  • model_path = models/best.pt: YOLO 가중치.
  • device = cuda:0: GPU 사용. GPU가 없으면 cpu 필요.
  • expected_image_width/height = 640/480: Pi 카메라 해상도와 반드시 일치.
  • intrinsic_file, handeye_result_json: 2D→3D 변환의 기반.

client_config.ini

  • grasp_server_url: 노트북의 현재 LAN IP와 포트 8000 사용. 기본 서버 주소는 192.168.0.11 기준으로 관리.
  • camera_stream fps = 4.0, timeout_sec = 0.8, jpeg_quality = 45: 미리보기 스트리밍 부하/화질.
  • camera_id: 0 또는 /dev/video2 같은 실제 장치 경로.
  • move_speed, move_timeout_sec: 로봇 이동 속도와 대기 시간.
  • random_motion: R 명령의 home 기준 x/y/z 랜덤 반경.
  • dry_run: 실제 로봇 명령 차단 여부. 첫 테스트는 true 권장.
  • show_window = false: Pi에 GUI DISPLAY가 없으면 false 유지.

8. 캘리브레이션 코드

marker.py는 수동 ChArUco 촬영 기반 캘리브레이션, auto_marker.py는 로봇이 여러 자세로 움직이며 자동으로 샘플을 수집하는 캘리브레이션 스크립트다. 결과물은 camera intrinsic과 eye-in-hand hand-eye transform이며, 서버의 geometry.py가 이 값을 사용한다.

  • camera_intrinsic_charuco.npz: 카메라 행렬 K와 distortion.
  • auto_handeye_result_*.json: 선택된 hand-eye method와 T_gripper_camera.
  • auto_handeye_charuco_samples_*.npz: 재검증 가능한 샘플 데이터.

9. 운영상 주의점과 개선 포인트

실제 로봇 동작 전: dry_run = true로 서버 응답, detection, flange_command, workspace gate 결과를 먼저 확인해야 한다.

자주 발생하는 문제

  • 서버 연결 timeout: Pi의 grasp_server_url이 노트북 현재 IP인지 확인. 같은 LAN이어야 한다.
  • Method Not Allowed: POST endpoint를 브라우저 주소창에서 연 경우다. 브라우저는 /camera-view만 연다.
  • OpenCV xcb/display 오류: Pi에서 로컬 창을 띄우려 한 경우다. show_window = false 유지.
  • Cannot open camera: v4l2-ctl --list-devices 또는 ls -l /dev/video*로 실제 capture 장치를 찾아 camera_id를 맞춘다.
  • 파지 좌표 이상: 캘리브레이션 파일, Euler order, 카메라 해상도, 테이블 plane Z를 우선 확인한다.

개선 포인트

  • 적용됨: 카메라 캡처를 producer thread + 최신 frame buffer 구조로 바꿔 메인 루프, 스트리밍, G 동작이 같은 VideoCapture.read()를 경쟁하지 않게 한다.
  • 적용됨: 실시간성 우선으로 save_results = false를 기본값으로 낮춰 grasp-plan 요청마다 raw/annotated/json을 쓰는 디스크 I/O를 줄인다.
  • 남은 선택 개선: Pi→server 프레임 전송도 persistent MJPEG/WebSocket으로 바꾸면 현재 프레임별 HTTP POST보다 지연과 부하를 더 줄일 수 있다.
  • 설정 필요: YOLO target class가 정해져 있으면 default_target_label을 설정해 잘못된 물체 선택을 줄인다.

11. 2026-07-03 수정사항 요약

  • 원격 명령을 /robot-command/next polling 방식에서 /robot-command/stream server-push 스트림 방식으로 변경. 웹 버튼을 눌러도 arm 클라이언트가 주기적으로 HTTP polling하지 않는다.

  • G 파지 후 표시되는 bbox/score overlay는 3초 뒤 자동으로 사라지도록 변경해 오래된 detection이 현재 결과처럼 남지 않게 했다.
  • 학습 실험용 dl_server/box.yaml과 dl_server/yolo_train.py를 Box.v1i.yolov11 데이터셋과 yolo11s.pt 기준으로 준비했다.
